# Repository Guidelines

## Project Structure & Module Organization

This repository is a small Vue 3 composable library written in TypeScript.

- `src/index.ts` is the package root export.
- `src/state/index.ts` exports state-related composables.
- `src/state/use*.ts` contains individual hooks, such as `useEditableList` and `useStorageAsync`.
- `src/**/*.test.ts` contains unit tests colocated with the source file they cover.
- `dist/` is generated by `pnpm build` and should not be edited manually.
- Build and test configuration lives in `tsup.config.ts`, `vitest.config.ts`, and `tsconfig.json`.

## Build, Test, and Development Commands

Use pnpm for all package operations.

```bash
pnpm install      # install dependencies from pnpm-lock.yaml
pnpm test         # run Vitest once
pnpm test:watch   # run Vitest in watch mode
pnpm typecheck    # run TypeScript without emitting files
pnpm build        # build ESM JS and .d.ts files with tsup
```

Run `pnpm test`, `pnpm typecheck`, and `pnpm build` before opening a PR.

## Coding Style & Naming Conventions

Write TypeScript using Vue 3 Composition API primitives. Prefer generic, reusable composables and explicit exported interfaces for public APIs.

- Use two-space indentation.
- Use single quotes and no semicolons, matching the existing files.
- Name composables with the `useXxx` pattern.
- Keep test files next to source files using `*.test.ts`.
- Use `.js` extensions in relative source exports, for example `export * from './useStorageAsync.js'`, so emitted ESM works in Node.

No formatter or linter is currently configured; keep changes consistent with nearby code.

## Testing Guidelines

Vitest is the test framework. Tests run in the Node environment and should focus on observable composable behavior, not implementation details.

For async Vue reactivity, use `await nextTick()` or await the exposed async method. Cover edge cases such as disabled states, max row limits, storage errors, and async race conditions.

Example:

```bash
pnpm test src/state/useStorageAsync.test.ts
```

## Commit & Pull Request Guidelines

There is no existing commit history to infer project-specific conventions. Use concise Conventional Commit-style messages, for example `feat: add storage hook tests` or `fix: prevent hydration overwrite`.

PRs should include:

- A short summary of the change.
- Any API changes or migration notes.
- Test evidence, usually `pnpm test`, `pnpm typecheck`, and `pnpm build`.
- Linked issues when applicable.

## Security & Configuration Tips

Do not commit secrets, local credentials, or generated dependency folders. `node_modules/` and `dist/` are generated artifacts; rebuild them locally when needed.
